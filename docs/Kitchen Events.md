# 🍳 KITCHEN EVENTS SYSTEM

Kitchen events fire randomly during gameplay. Each event requires chat to respond collectively — progress scales with participation.

---

## Architecture

| File | Role |
|------|------|
| `src/hooks/useKitchenEvents.ts` | Spawn timer, command handling, resolution logic, audio triggers |
| `src/data/kitchenEventDefs.ts` | Event definitions, command pools, constants, generator functions |
| `src/state/types.ts` | `KitchenEvent` interface, `EventType` / `EventCategory` unions |
| `src/state/gameReducer.ts` | Event effect actions (speed modifier, money multiplier, ingredients, stations, patience) |
| `src/components/EventCardOverlay.tsx` | Receipt-ticket overlay with per-event colours, time bar, progress bar, stamp + tear-off exit |

---

## Event Categories

### Hazard — Penalty (`hazard-penalty`)
Time-limited. If threshold isn't met in **10 s**, a negative consequence is applied.

### Hazard — Immediate (`hazard-immediate`)
No time limit. Negative effect is *active* until threshold is met — resolving it removes the effect.

### Opportunity (`opportunity`)
Time-limited. If threshold is met in **12 s**, a reward is granted.

---

## Participation Threshold

All events use:
```
threshold = ceil(activePlayerCount × 0.8), minimum 1
```
Each user can only contribute once per event (tracked in `respondedUsers`).

### 🐀 Rat Invasion — `rat_invasion`
**Category:** Hazard — Penalty

| Field | Value |
|-------|-------|
| Command (randomly picked) | `SHOO SHOO SHOO`, `CHASE CHASE CHASE`, or `BEGONE BEGONE BEGONE` |
| Time limit | 10 s |
| **On fail** | Removes 3 random `preparedItems` from inventory (`REMOVE_PREPARED_ITEMS`) |
| **On resolve** | No effect |

### 🔌 Power Trip — `power_trip`
**Category:** Hazard — Immediate

| Field | Value |
|-------|-------|
| Command | Displays a randomly generated **math equation** — players type the numeric answer (e.g. `42`) |
| Time limit | None — effect persists until resolved |
| **Active effect** | 2 random non-overheated stations disabled (`DISABLE_STATIONS`) |
| **On resolve** | Stations re-enabled (`ENABLE_STATIONS`) |
| Spawn condition | Requires ≥ 2 non-overheated stations |

The equation is generated by `makePowerTripEquation()` and produces add/subtract (3 operands) or multiply/divide problems. The numeric answer is stored in `payload.powerTripAnswer`; matching compares `normalized === String(payload.powerTripAnswer)`.

### 🧩 Mystery Recipe — `mystery_recipe`
**Category:** Opportunity

| Field | Value |
|-------|-------|
| Command | Anagram of a random ingredient name from the enabled recipe set |
| Time limit | 12 s |
| **On resolve** | Adds 3 random produced ingredients to `preparedItems` (`ADD_PREPARED_ITEMS`) |
| **On fail** | No reward |

The answer is a random step-target from `getIngredientTargets(enabledRecipes)` with underscores replaced by spaces. The anagram is generated by `makeAnagram()`. Answer stored in `payload.anagramAnswer`; matching compares `normalized === payload.anagramAnswer.toUpperCase()`.

### 📶 Wifi Password — `typing_frenzy`
**Category:** Opportunity

| Field | Value |
|-------|-------|
| Command | A randomly generated **15-character gibberish string** (alphanumeric + `!@#$%^&*+=`; first char always alphanumeric) |
| Time limit | 12 s |
| **On resolve** | `moneyMultiplier = 1.5` for **20 s** (`ADD_MONEY_MULTIPLIER`) |
| **On fail** | No reward |

Generated by `makeTypingFrenzyPhrase()`. Stored in `payload.typingPhrase`; matching uses `ev.chosenCommand` directly.

---

### 🕺 Dance — `dance`
**Category:** Opportunity

| Field | Value |
|-------|-------|
| Command | The full 4-direction sequence typed as one message, e.g. `UP DOWN RIGHT RIGHT` |
| Time limit | 12 s |
| **On resolve** | All active orders gain **+15 s** patience (`EXTEND_ORDER_PATIENCE`) |
| **On fail** | No reward |

**Mechanic:**
1. A random 4-step sequence is generated at spawn by `makeDanceSequence()` (e.g. `['UP', 'DOWN', 'RIGHT', 'RIGHT']`). Directions can repeat.
2. `chosenCommand` is set to the space-joined sequence (e.g. `"UP DOWN RIGHT RIGHT"`). The sequence is **not** announced in chat — players must read the card.
3. **Memorise phase (first 3 s):** The card shows all 4 directional arrows simultaneously.
4. **Type phase (remaining 9 s):** Arrows flip to `?` boxes. Players type the full sequence from memory as a single chat message.
5. Progress uses the standard `respondedUsers` mechanism — each player can contribute once.

Stored in `payload.danceSequence`. Matching compares `normalized === ev.chosenCommand`.

## Spawn Logic

- Spawn interval: randomised between `kitchenEventSpawnMin` and `kitchenEventSpawnMax` (seconds); reset each time an event ends.
- Timer only ticks when no event is active and the game is not paused.
- Back-to-back identical event types are avoided (falls back to full pool if only one type is enabled).
- Power Trip skips spawn if fewer than 2 non-overheated stations exist.
- Spawning is disabled when `active = false` (game not playing, tutorial active, or `kitchenEventsEnabled = false`).

## Settings (`GameOptions`)

| Option | Default | Description |
|--------|---------|-------------|
| `kitchenEventsEnabled` | `true` | Master toggle for the entire system |
| `enabledKitchenEvents` | `[]` (all) | Restrict pool to specific event types; empty = all enabled |
| `kitchenEventSpawnMin` | `30` | Minimum seconds between spawns |
| `kitchenEventSpawnMax` | `60` | Maximum seconds between spawns |

## UI — EventCardOverlay

The overlay uses a **receipt ticket** theme: a metal clip at the top, coloured gradient header, body with command box and progress bars, perforated bottom edge.

- **Entry:** Ticket prints down from clip on spawn (CSS `clip-path` + bounce animation). Screen flashes red (hazard) or green (opportunity).
- **Active:** Pulsing vignette around the screen edges while the event is live.
- **Dance card:** Shows all arrows during the memorise phase (with gentle pulse), then `?` boxes during the type phase. Label switches from "🧠 Memorise!" to "🕺 Type the full sequence!".
- **Bars:** Striped time bar (event accent colour) + green progress bar. Dance shows Progress (respondedUsers / threshold).
- **Conclusion:** Rubber stamp animates in (green ✓ RESOLVED or red ✗ FAILED), then ticket tears upward off screen ~900 ms later. `setActiveEvent(null)` fires after 1 500 ms.
- **Double-resolve guard:** `concludingEventIdRef` prevents resolve/fail being called twice if React hasn't re-rendered.

### Audio

Each event def has an `audio` field: `{ ambient, success, fail? }`. On spawn, `getAudioManager().startEventAmbient(ambient)` plays a looping ambient. On resolve/fail, `stopEventAmbient()` then `playEventSfx(success/fail)` (500 ms delay).

## Command Matching

```
Twitch Chat / Local Input
  → App.tsx handleTwitchMessage / handleChatSend
  → useKitchenEvents.handleEventCommand(user, text)
      normalized = text.trim().toUpperCase()
      mystery_recipe → compare normalized to payload.anagramAnswer.toUpperCase()
      power_trip     → compare normalized to String(payload.powerTripAnswer)
      all others     → compare normalized to ev.chosenCommand
      deduplication  → ev.respondedUsers.includes(user)
  → on threshold met: dispatch effect action + mark resolved
  → on time expiry:  dispatch penalty action (if any) + mark failed
```
